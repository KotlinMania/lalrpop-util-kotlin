# lalrpop-util-kotlin

[![Discord](https://img.shields.io/badge/Discord-Solace%20Project-5865F2?logo=discord&logoColor=white)](https://discord.gg/rJqVeSmx4)

Kotlin Multiplatform port of the [`lalrpop-util`](https://crates.io/crates/lalrpop-util) crate — the small runtime support library every parser generated by `lalrpop` (or its Kotlin port [`lalrpop-kotlin`](https://github.com/KotlinMania/lalrpop-kotlin)) links against.

This artifact is the runtime contract. It is intentionally separate from the parser-generator itself so consumers of generated parsers do not have to ship the entire generator pipeline at runtime.

Maintained by Sydney Renee &lt;sydney@solace.ofharmony.ai&gt;, The Solace Project (KotlinMania org).

## What's inside

The runtime API mirrors upstream Rust `lalrpop_util`:

| Upstream Rust | Kotlin |
|---|---|
| `lalrpop_util::ParseError<L, T, E>` | `io.github.kotlinmania.lalrpoputil.ParseError<L, T, E>` |
| `lalrpop_util::ErrorRecovery<L, T, E>` | `io.github.kotlinmania.lalrpoputil.ErrorRecovery<L, T, E>` |
| `lalrpop_util::state_machine::ParserDefinition` | `…lalrpoputil.statemachine.ParserDefinition` |
| `lalrpop_util::state_machine::ParserAction` | `…lalrpoputil.statemachine.ParserAction` |
| `lalrpop_util::state_machine::SimulatedReduce` | `…lalrpoputil.statemachine.SimulatedReduce` |
| `lalrpop_util::state_machine::Parser<D, I>` + `drive(...)` | `…lalrpoputil.statemachine.Parser<…>` + `Parser.Companion.drive(...)` |
| `lalrpop_util::lexer::Token<'input>` | `…lalrpoputil.lexer.Token` *(planned)* |
| `lalrpop_util::lexer::MatcherBuilder` | `…lalrpoputil.lexer.MatcherBuilder` *(planned)* |
| `lalrpop_util::lexer::Matcher` | `…lalrpoputil.lexer.Matcher` *(planned)* |

Plus a Kotlin-only adapter layer (`Production`, `ParseTables`, `TableDrivenParserDefinition`, `TableDrivenLr1Driver`, `ParseStack`) that the Kotlin-emit codegen back-end builds on top of. Upstream Rust embeds the same data inline as `const __ACTION: &[i16] = &[…]` arrays in the generated source; on the Kotlin side it is more idiomatic to ship those as `IntArray` data exposed through a table-driven `ParserDefinition` implementation.

## Targets

Kotlin Multiplatform, no JVM. Shipped:

- `macosArm64`, `macosX64`
- `linuxX64`
- `mingwX64`
- `iosArm64`, `iosX64`, `iosSimulatorArm64`
- `js` (browser + nodejs)
- `wasmJs` (browser + nodejs)
- `androidLibrary`

## Coordinates

```kotlin
dependencies {
    implementation("io.github.kotlinmania:lalrpop-util-kotlin:0.1.0")
}
```

The version stays in lockstep with the upstream `lalrpop-util` crate it tracks (currently `0.23.x`). The Kotlin artifact's first patch digit is its own line; the major/minor follow upstream's API surface.

## Building from source

```bash
git clone https://github.com/KotlinMania/lalrpop-util-kotlin.git
cd lalrpop-util-kotlin
./setup-android-sdk.sh        # one-shot, gitignored .android-sdk/
./gradlew test
```

`setup-android-sdk.sh` downloads Google's command-line tools, accepts SDK licenses non-interactively, and stages everything under `./.android-sdk/`. The Android target is the only one that needs it; the rest of the targets build without.

The build gate is `./gradlew test`. There is no JVM target.

## Relationship to lalrpop-kotlin

```
┌────────────────────────────────────────┐
│ your-grammar.lalrpop                   │
└──────────────┬─────────────────────────┘
               │ Gradle codegen task at build time, calling
               ▼
┌────────────────────────────────────────┐
│ lalrpop-kotlin                         │  ← generator (build-time only)
│ (parser, normalizer, LR builder,       │
│  Rust-emit + Kotlin-emit backends)     │
└──────────────┬─────────────────────────┘
               │ emits StarlarkParser.kt (or your-grammar.kt)
               ▼
┌────────────────────────────────────────┐
│ your-grammar.kt + your-app             │
└──────────────┬─────────────────────────┘
               │ implementation(...)
               ▼
┌────────────────────────────────────────┐
│ lalrpop-util-kotlin (this project)     │  ← runtime (linked at app build time)
│   ParseError, ParserDefinition,        │
│   table-driven LR driver, drive()      │
└────────────────────────────────────────┘
```

The split mirrors upstream Rust's `lalrpop` build-dep + `lalrpop-util` regular-dep distinction.
